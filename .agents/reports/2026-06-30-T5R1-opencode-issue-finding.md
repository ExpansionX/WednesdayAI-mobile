## Prose Justification

### Finding 1 — Unconditional `useGatewayRuntimeSettings` work for YouMind
`ModelsScreen` calls `useGatewayRuntimeSettings({ gateway, gatewayEpoch, hasActiveGateway })` unconditionally (line 24-28) and builds the `modelConfig` memo (line 30-54) for *every* backend, then dispatches the body via `selectByBackend`. For YouMind the body is a static `EmptyState`, so none of `modelConfig`'s fields are consumed. The hook internally derives `capabilities` from config and gates `loadGatewaySettings`/`saveGatewaySettings` on `capabilities.configRead`/`configWrite` (both false for YouMind per gateway-backends.ts:118-119), so it should short-circuit before any network call — but the hook still allocates state, runs effects, and computes derived values that ModelsScreen throws away. This is not a correctness bug; it is over-eager work that the AGENTS.md "capability-gated" guidance would prefer to short-circuit at the screen level. Severity low because the hook's capability gating prevents actual network I/O for YouMind.

### Finding 2 — `selectByBackend` youmind fallback latent risk
`selectByBackend` (gateway-backends.ts:226-237) returns `options.youmind ?? options.openclaw` when `kind === 'youmind'`. Before this diff, ModelsScreen's youmind branch (per the prompt) rendered a fully-proped `ModelsView`, i.e. it relied on the *implicit* OpenClaw-fallback behavior in callers that omitted `youmind`. After this diff, ModelsScreen passes an explicit `youmind: <EmptyState>`, so the fallback is no longer exercised here. The change is safe for ModelsScreen. I flag the latent risk because the diff's intent ("model selection is not available for this backend") contradicts the prior implicit contract that `youmind → openclaw` for model UI. If a future caller omits `youmind` expecting OpenClaw-like model UI, they will get the fallback — which is now *inconsistent* with ModelsScreen's explicit stance. Not a current bug; a documentation/contract drift worth noting.

### Finding 3 — `deriveBaseUrl` catch comment accuracy
The new comment is dense and mostly correct. The one edge it glosses over is a URL that fails `new URL()` *only* because of a malformed query string. JavaScript's `URL` constructor is lenient on query/fragment syntax (it does not throw on `?token=abc#bad[`), so the try path's `url.search = ''; url.hash = '';` already neutralizes those. I could not construct an input where the old catch's `split('?')[0].split('#')[0]` would have succeeded and the new catch's `return null` would have wrongly discarded a recoverable URL. The existing tests (`gateway-backend-operations.test.ts:120-134`) explicitly cover the malformed-host and bare-protocol catch paths and assert `null`, matching the new behavior. So the comment's "always returns null" claim is empirically true for reachable inputs. The nit is that the comment's *reasoning* ("stripping query/hash/path cannot fix a bad host") is correct but understates that the try path already handles the query/hash cases that *would* have been recoverable. Low severity.

### Findings 4 & 5 & 6 & 10 — Test coverage gaps (no behavior bugs)
The three new tests are well-constructed and prove their stated claims:
- `config.get` retries on `[BRIDGE_UNAVAILABLE]` (1 retry asserted; 2nd retry not asserted for config.get specifically, but the shared delay array is pinned by the `sessions.list` test).
- `agents.files.get` does NOT auto-retry (1 call asserted).
- `HERMES_BRIDGE_RETRY_METHODS ⊆ shouldTraceRequest` (13-method subset verified).

The gaps are: (a) no assertion that retry re-sends identical `params` (Finding 4); (b) no test pinning the *reason* `config.get` is safe-to-retry while `agents.files.get` is not — i.e. the `baseHash` staleness contract on the write side (Finding 5); (c) the subset test does not guard the converse direction (Finding 6, which is the *safe* direction anyway); (d) `config.get`'s second retry is not exercised (Finding 10). None of these are behavior bugs; they are coverage rigor improvements. The production code under test is correct.

### Finding 7 — Defense-in-depth confirmation
`getAgentFile`'s `'File not found'` throw is excluded from retry by two independent gates: method-set membership (`HERMES_BRIDGE_RETRY_METHODS` does not contain `agents.files.get`) and error-message matching (`isHermesBridgeUnavailableError` checks for the `[BRIDGE_UNAVAILABLE]` substring, which `'File not found'` lacks). Either gate alone would suffice. No issue.

### Finding 8 — Pre-existing YouMind files-path exposure (out of diff)
`sharedOperations.getAgentFile` is inherited by `YOUMIND_OPERATIONS` (gateway-backend-operations.ts:275-277 spreads `OPENCLAW_OPERATIONS`). `gateway.getAgentFile` (gateway.ts:1040-1041) calls it via `sendBackendRequest`. For a YouMind config, `sendBackendRequest` → `sendRequestWithHermesBridgeRetry` → `isHermesRelayBridgeRetryEligible` returns false (backend kind is `youmind`, not `hermes`), so no retry; but the request itself would still be dispatched over a transport YouMind does not have (`YOUMIND_CAPABILITIES.gatewayConnection = false`). This is pre-existing, not introduced by this diff, and is mitigated by capability-gated UI hiding the Files entry. Flagged only because the diff widens the YouMind surface (ModelsScreen now explicitly branches for YouMind) while leaving this adjacent path unaddressed.

### Finding 9 — i18n correctness
Verified via grep: `"Model selection is not available for this backend."` exists in all six locale `console.json` files (en:1051, de:1028, es:1028, ja:1028, ko:1028, zh-Hans:1051). The en value equals the key. Translations are natural and well-formed. `"No models available"` was already present in all six `console.json` files before this diff (line 173 in each), so the diff correctly reused the existing key rather than re-adding it. No orphans, no missing locales. Fully compliant with AGENTS.md i18n rules.

### Finding 11 — YouMind `EmptyState` vertical centering
`EmptyState`'s root style is `flex: 1; alignItems: center; justifyContent: center` (EmptyState.tsx:32-38). When rendered as the sole child of ModelsScreen's body for YouMind, it will vertically center the "No models available" message. AGENTS.md "First-Screen Rhythm Rules" prefers list/card content to start at a standard first-section offset (`Space.sm`/`Space.md`), not center. For a true "this feature is unavailable" empty state, vertical centering is arguably appropriate, so this is a minor visual-consistency concern, not a bug.

### Finding 12 — `wsPathPattern` still used
Confirming no dead-parameter regression: `wsPathPattern` is still consumed in the try path (gateway-backend-operations.ts:293, `url.pathname.replace(wsPathPattern, '')`). The catch simplification removed the only *catch-path* use of the parameter, but the try-path use remains. No dead code introduced.

## Summary

No critical, high, or medium-severity bugs were found in the T3+T4 remediation diff. The `deriveBaseUrl` catch simplification is behavior-preserving for all reachable inputs (verified against existing tests). The three new gateway tests prove their stated claims. i18n is complete across all six locales. The YouMind `EmptyState` swap in ModelsScreen is safe and does not regress OpenClaw or Hermes paths.

The findings are low-severity quality observations: over-eager hook execution for YouMind (Finding 1), a latent `selectByBackend` contract drift (Finding 2), a comment-accuracy nuance (Finding 3), four test-rigor gaps (Findings 4, 5, 6, 10), one pre-existing out-of-diff exposure (Finding 8), and one minor visual-consistency note (Finding 11). None block merge; all are optional improvements.

Confidence: high that no behavior bug exists in the diff; medium on the latent risks (Findings 2 and 8) since they depend on future caller behavior and pre-existing wiring not modified by this diff.
